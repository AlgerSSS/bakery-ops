# Hot Crush 门店智能复盘系统

## 触发条件

店长在 WhatsApp 发送包含以下关键词的消息时触发：
- 复盘、日复盘、今日复盘、今天总结、daily review
- 或消息中包含≥3个复盘特征词：业绩、客单、报废、断货、人效、时段、机会点

## 复盘流程

```
店长发消息 → 意图识别为"复盘"
    ↓
LLM 解析店长输入 → 提取结构化信息（日期/特殊事件/报废/国籍等）
    ↓
数据查询层（并行）
├─ daily_revenue → 当日营业额/客单数/客单价/折扣率/会员占比
├─ hourly_sales_summary → 每小时客单数/客单价/营业额
├─ item_hourly_sales → 每小时每单品销量（断货分析用）
├─ daily_payment_breakdown → 支付渠道占比
├─ daily_dining_breakdown → 堂食/外带
├─ daily_pnl → 报废/成本/利润（损益表）
├─ forecast → 今日应做目标 + 达成率
├─ daily_revenue(上周同天) → 周同比
└─ RAG → 往期复盘追踪 + SOP + 历史规则
    ↓
LLM 生成分析报告（见输出格式）
    ↓
多轮对话（店长追问 → 查数据库 → 回答）
    ↓
对话结束 → 提炼知识 → 写入 RAG
```

## 数据路由逻辑

### 必查（每次复盘都执行）

| 数据 | 来源 | 用途 |
|------|------|------|
| 当日营业额/客单数/客单价 | daily_revenue | 核心KPI |
| 折扣率 | daily_revenue.discount_rate | 促销效果 |
| 会员占比 | daily_revenue.member_sales_ratio | 客群忠诚度 |
| 今日应做 | forecast系统 | 达成率计算 |
| 时段明细 | hourly_sales_summary | 高峰/低谷分析 |
| 单品TOP15 | item_hourly_sales | 爆品/滞销 |
| 上周同天 | daily_revenue | 周同比 |
| 支付渠道 | daily_payment_breakdown | 客群画像 |
| 往期复盘追踪 | RAG | 建议是否落地 |

### 条件查（异常时追加）

| 触发条件 | 额外查询 | 目的 |
|----------|----------|------|
| 报废率 > 5% | RAG: 历史报废原因+改善方案 | 找规律 |
| 达成率 < 80% | 同星期历史均值 | 判断是否异常 |
| 品尝报废 > RM1,452/天 | RAG: 品控SOP | 管控建议 |
| 前场人效 < RM350/人·时 | RAG: 排班优化方案 | 人效改善 |
| 店长提到断货 | item_hourly_sales 断货前后对比 | 损失估算 |
| 店长提到天气/活动 | RAG: 类似情况历史表现 | 归因分析 |

## 输出格式

LLM 动态生成，非固定模板。核心要求：

1. 开头"核心发现"直接亮最重要的1-2个结论
2. 数字必须从数据库取，标注来源
3. 必须有 vs 上周同天的对比
4. 必须有今日应做达成率
5. 必须有时段分析（高峰/低谷 + 原因）
6. 必须有调整追踪：上次建议→执行效果（从RAG获取）
7. 建议≤3条，基于数据，具体到人/时段/品类
8. 语言风格：犀利、专业、数据驱动，像经验丰富的运营总监

## 多轮对话

店长可以追问任何数据问题，系统会：
1. LLM 判断意图（问哪个产品/时段/日期）
2. 查对应数据库表
3. 用数据回答

结束信号：店长说"没了"/"好的"/"结束" → 提炼本次对话知识 → 写入RAG

## 知识沉淀

对话结束时自动提取：
- 确认的运营决策（如：蛋挞备货量+20%）
- 发现的规律（如：下雨天客流-30%）
- 待验证假设（如：对面新店分流年轻客群）

写入 LightRAG，下次复盘时作为"往期追踪"参考。

## Pitfalls

- 数据以数据库为准，店长报告的数字仅作参考对比
- 报废数据来自爬取数据
- 人效需要班表数据（daily_schedule），目前暂缺，等接入后自动计算
- 今日应做来自 forecast 系统，如果 business_rules 未配置会跳过
- RAG 可能返回过期信息，LLM 应以数据库数字为准
- 30天滚动数据每日凌晨1点自动更新，复盘时数据是截至上次刷新的
