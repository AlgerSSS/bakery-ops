# Hot Crush 门店智能复盘系统

## 触发条件

店长在 WhatsApp 发送包含以下关键词的消息时触发：
- 复盘、日复盘、今日复盘、今天总结、daily review
- 或消息中包含≥3个复盘特征词：业绩、客单、报废、断货、人效、时段、机会点

## 复盘流程

```
店长发消息 → 意图识别为"复盘"
    ↓
LLM 解析店长输入 → 提取结构化信息（日期/特殊事件/报废/国籍等）
    ↓
数据查询层（并行）
├─ POS销售系统 → 当日营业额/客单数/客单价/折扣率/会员占比
├─ 分时段销售 → 每小时客单数/客单价/营业额
├─ 单品销售明细 → 每小时每单品销量（断货分析用）
├─ 支付渠道数据 → 支付渠道占比
├─ 堂食外带数据 → 堂食/外带
├─ 损益表 → 报废/成本/利润
├─ 排产目标 → 今日应做目标 + 达成率
├─ POS销售系统(上周同天) → 周同比
└─ 历史经验库 → 往期复盘追踪 + SOP + 历史规则
    ↓
LLM 生成分析报告（见输出格式）
    ↓
多轮对话（店长追问 → 查数据库 → 回答）
    ↓
对话结束 → 提炼知识 → 写入历史经验库
```

## 数据路由逻辑

### 必查（每次复盘都执行）

| 数据 | 来源 | 用途 |
|------|------|------|
| 当日营业额/客单数/客单价 | POS销售系统 | 核心KPI |
| 折扣率 | POS销售系统 | 促销效果 |
| 会员占比 | POS销售系统 | 客群忠诚度 |
| 今日应做 | 排产目标 | 达成率计算 |
| 时段明细 | 分时段销售 | 高峰/低谷分析 |
| 单品TOP15 | 单品销售明细 | 爆品/滞销 |
| 上周同天 | POS销售系统 | 周同比 |
| 支付渠道 | 支付渠道数据 | 客群画像 |
| 往期复盘追踪 | 历史经验库 | 建议是否落地 |

### 条件查（异常时追加）

| 触发条件 | 额外查询 | 目的 |
|----------|----------|------|
| 报废率 > 5% | 历史经验库: 历史报废原因+改善方案 | 找规律 |
| 达成率 < 80% | 同星期历史均值 | 判断是否异常 |
| 品尝报废 > RM1,452/天 | 历史经验库: 品控SOP | 管控建议 |
| 前场人效 < RM350/人·时 | 历史经验库: 排班优化方案 | 人效改善 |
| 店长提到断货 | 单品销售明细 断货前后对比 | 损失估算 |
| 店长提到天气/活动 | 历史经验库: 类似情况历史表现 | 归因分析 |

## 输出格式

输出会渲染成 Lark 卡片，**必须严格用下面的 Markdown 结构**（## 会渲染成加粗小标题，表格会渲染成对齐分栏，--- 会渲染成分隔线，**加粗**会真加粗）。核心要求：

1. 开头"核心发现"用 1-2 句亮最重要结论，**关键数字加粗**
2. 核心指标**必须用 Markdown 表格**（含 |---| 分隔行）做「本期 vs 上周同天 vs 变化」三列对比——**必须含营业额(应收)、实收(折后)、水吧营业额**三项
3. 必须有今日应做达成率
4. 时段分析：高峰/低谷**只在 12:00–22:00 营业时段内取**（用数据里的「营业时段峰谷」，打烊尾单不算低谷）+ 一句趋势归因
5. 单品表现 TOP5：**按销量排**、用中文品名、**必须含 TC占比**（每单渗透率）
6. 报废分析：报废率 + 三分类；**报废王只取「排产报废」**（可改进的过量）；**试吃(品尝)单列为"投入"，不进报废王**
7. **品类结构**：各品类营业额占比+件均价，一句点出拉高/拉低客单价的品类
8. **毛利与损失**：低毛利高量隐患品 + 排产报废真实成本损失 + **断货损失额**（断货损失用于解释达成率缺口；数据里有「断货损失」才写）
9. **预估 vs 实卖**：只做归因（真过量 vs 断货少卖），**绝不输出"减产N个/加产N个"指令**
10. 客群与支付：会员占比、外带占比、支付结构
11. 明日行动 ≤3 条，具体到人/时段/品类，标优先级；**只给方向，不给精确加减产数字**
12. 有历史经验时加一句"上次建议→执行效果"追踪
10. 数字以系统数据为准；语言犀利、专业、像运营总监
11. 各大板块之间用 `---` 分隔

⚠️【硬规则】① 输出给店长看，**绝对不能出现任何英文数据库表名或技术字段**（如 daily_revenue、hourly_sales_summary、item_hourly_sales、daily_pnl、item_waste、forecast、RAG）；来源只用中文业务名称。② 某类数据确实没有时，那一小节整块省略，**不要输出"暂无数据"占位**。

## 多轮对话

店长可以追问任何数据问题，系统会：
1. LLM 判断意图（问哪个产品/时段/日期）
2. 查对应数据库表
3. 用数据回答

结束信号：店长说"没了"/"好的"/"结束" → 提炼本次对话知识 → 写入历史经验库

## 知识沉淀

对话结束时自动提取：
- 确认的运营决策（如：蛋挞备货量+20%）
- 发现的规律（如：下雨天客流-30%）
- 待验证假设（如：对面新店分流年轻客群）

写入历史经验库，下次复盘时作为"往期追踪"参考。

## Pitfalls

- 数据以数据库为准，店长报告的数字仅作参考对比
- 报废数据来自爬取数据
- 人效需要班表数据（排班表），目前暂缺，等接入后自动计算
- 今日应做来自排产目标，如果经营规则未配置会跳过
- 历史经验库可能返回过期信息，应以系统实时数字为准
- 30天滚动数据每日凌晨1点自动更新，复盘时数据是截至上次刷新的
